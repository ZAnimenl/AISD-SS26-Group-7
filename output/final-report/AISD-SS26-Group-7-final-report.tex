\section{User Guide}

\subsection{Purpose of the Platform}

The platform is a browser-based coding assessment system for practical software development tasks. It supports two main user groups. Administrators create assessments, review and edit AI-generated task drafts, manage users, and inspect results. Students register or sign in, start assigned assessments, work in an IDE-like browser workspace, use a bounded AI assistant, submit their work, and complete a short reflection on their AI usage.

The system is currently intended for local demonstration and course evaluation. The team plans to move it from localhost to an online deployment in late July. Until then, all setup steps below assume a local development environment.

\subsection{Local Setup}

The local setup follows the project README and is intentionally kept to one main command. A new user should install Node.js 20 or newer, npm, the .NET SDK compatible with the backend solution, and optionally a Docker-compatible runtime for code execution and final submission.

\begin{enumerate}
  \item Clone the repository:
  \begin{verbatim}
git clone https://github.com/ZAnimenl/AISD-SS26-Group-7.git
cd AISD-SS26-Group-7
  \end{verbatim}
  \item Start the full local application:
  \begin{verbatim}
npm run dev
  \end{verbatim}
  \item Open the frontend at:
  \begin{verbatim}
http://localhost:3000
  \end{verbatim}
  \item The backend API runs locally at:
  \begin{verbatim}
http://localhost:5140/api/v1
  \end{verbatim}
\end{enumerate}

The startup script prepares a local SQLite database under \texttt{.local-data/}, loads \texttt{.env.local}, restores dependencies when needed, starts the backend, waits for the health endpoint, and then starts the Next.js frontend. Docker is only required for real run/submit grading. Without Docker, the app still starts, but the workspace explains that final submit is unavailable until the sandbox runtime can be reached.

\subsubsection{Creating and Managing Assessments}

Administrators use the assessment section to create, edit, and delete assignments. An assessment has a title, description, duration, availability window, publication status, AI-enabled flag, and one or more practical questions. The current task categories are:

\begin{itemize}
  \item frontend UI extension,
  \item REST API development,
  \item database query/schema work,
  \item bug fixing in existing code.
\end{itemize}

The create flow is split into steps so that administrators first define the assessment, then generate or review questions, and finally confirm delivery settings. AI-generated drafts are treated as drafts: the administrator is expected to review the task text, starter files, visible and hidden tests, metadata, and scoring configuration before publishing.

The report view shows assessment-level summaries and supports drill-down into student evidence. \subsubsection{Assessment Workspace}

The workspace is the main student experience. It is designed like a compact IDE:

\begin{itemize}

  \item The bottom output panel contains preview and console tabs for run feedback.
\end{itemize}

The frontend does not execute student code locally. Run and submit go through backend assessment-scoped APIs, and the backend resolves the active attempt from the authenticated user and assessment ID. \subsubsection{AI Assistance and Reflection}

When AI is enabled, students can ask for code suggestions, explanations, or debugging help. The assistant receives only visible task context, visible workspace files, selected language, active file, and public run output. Hidden tests, grading implementation, provider keys, and server-side prompts remain on the backend.

After submission, students complete a short reflection of at most 100 words. The reflection asks how the AI assistant was used, which suggestion helped and how it was verified, and which suggestion was rejected, corrected, or unhelpful. The reflection has a backend-owned timer and is included in AI usage grading.

\subsubsection{Results}

After the assessment is submitted and any required AI-usage grading is complete, students can review results. The student result page shows functional score, AI usage score when applicable, final score, reflection status, and a summary of feedback. Administrators see a richer version in reports, including evidence and retry controls.

\section{Technical Documentation}

\subsection{System Overview}

The project addresses a practical problem in modern software-engineering education: students increasingly use AI tools while solving development tasks, but many assessment systems still grade only final code output. Our platform grades the submitted code and also records how the embedded AI assistant was used. The goal is not to punish AI use. Instead, the system makes AI usage visible, auditable, and assessable as part of a realistic development workflow.

The scope of the implemented MVP includes:

\begin{itemize}
  \item administrator-managed assessments, questions, test cases, and users,
  \item student registration, login, dashboard, assessment start, workspace, reflection, and results,
  \item backend-owned attempts and workspace persistence,
  \item Docker-based run/submit architecture for public and hidden tests,
  \item embedded AI assistance through a backend AI layer,
  \item AI interaction logging, token tracking, reflection, and automatic AI usage scoring,
  \item reporting for functional, AI-usage, and final scores.
\end{itemize}

The clean architecture view in Figure~\ref{fig:arch-clean} shows the main deployment shape. Students and administrators interact with a Next.js web application. The frontend sends API requests to an ASP.NET Core backend. The backend owns authentication, assessment state, workspace persistence, AI integration, grading, reporting, and database access. External services are connected only through the backend.

Persistent state is stored through EF Core. Code execution is isolated through a Docker code runner. AI completion calls go through the backend AI service and then to the DeepSeek provider.

\subsection{AI Architecture}

The AI layer has three responsibilities. First, it provides student assistance in the workspace. Second, it supports administrators by generating assessment or question drafts for review. Third, it grades AI usage after submission using interaction logs, token data, events, execution history, and the student's reflection.

The clean AI workflow is shown in Figure~\ref{fig:ai-clean}. Both student and administrator actions enter through backend endpoints. The backend authenticates the user, checks the role, builds a safe context, selects the correct AI workflow, requests a structured completion when needed, and stores the resulting transcript, tokens, events, scores, or draft content in the database.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\linewidth]{\figpath/ai-workflow-clean.png}
  \caption{Clean AI workflow across student assistance, admin generation, and grading.}
  \label{fig:ai-clean}
\end{figure}

The student detailed workflow in Figure~\ref{fig:ai-student} shows the security boundary. The frontend sends the selected language, active file, visible files, message, and last public run result. The backend resolves the active attempt from the authenticated student and assessment ID. It does not trust or accept frontend-owned session IDs. The AI assistant service excludes hidden tests, grading logic, provider keys, and server-only prompts before calling the completion service. After the response is parsed, the backend stores the interaction and token usage, and the frontend displays the markdown answer and safe workspace actions.

\begin{figure}[H]
  \centering
  \includegraphics[angle=90,height=0.76\textheight]{\figpath/ai-workflow-student-detailed.png}
  \caption{Detailed student AI assistance workflow.}

\end{figure}

The admin detailed workflow in Figure~\ref{fig:ai-admin} shows two administrative AI flows. In the draft-generation flow, the admin requests an assessment or question draft. The backend validates the administrator role, builds a prompt from prototype, task type, difficulty, and language constraints, calls the provider through the completion service, parses the draft, and stores questions, starter files, test cases, and metadata for administrator review. In the reporting flow, the admin can request or retry AI-usage grading. The grading service loads submissions, transcripts, token usage, and reflection data, optionally calls the provider, and stores the score, summary, confidence, and details.

\subsection{Workflow Implementation and Trade-offs}

We chose a backend-owned AI workflow because it keeps sensitive material out of the browser. This made the implementation more complex than a direct frontend-to-provider call, but it gave us stronger control over role checks, prompt construction, token tracking, logging, and failure handling.

The main advantages are:

\begin{itemize}
  \item hidden tests and grading details never enter the student UI,
  \item AI provider keys stay server-side,
  \item every AI interaction can be logged with token usage and semantic tags,
  \item administrator review remains part of generated task publication,
  \item AI usage grading has access to a full attempt history rather than a single prompt.
\end{itemize}

The main costs are:

\begin{itemize}
  \item more backend contract work is needed for every frontend AI feature,
  \item provider errors must be represented clearly to users,
  \item generating good tasks requires strict validation and correction loops,
  \item local demonstrations depend on real external configuration for AI and Docker.
\end{itemize}

\section{Project Management}

\subsection{Project Vision}

Our vision was to build an assessment platform that reflects how coding is now done in practice. Students should be able to solve realistic software tasks in the browser, use an embedded AI assistant, and still be assessed fairly. Administrators should be able to create tasks, review AI-generated drafts, manage students, and inspect both code correctness and AI usage.

The key deliverables were:

\begin{itemize}
  \item a working Next.js frontend with student and administrator workflows,
  \item an ASP.NET Core backend with authentication, RBAC, assessments, attempts, workspace state, reports, and AI endpoints,
  \item a sandboxed grading architecture based on Docker,
  \item AI assistance, task generation, transcript logging, token tracking, and AI usage scoring,
  \item documentation, diagrams, startup tooling, and verification tests.
\end{itemize}

The boundaries were also important. We did not try to build a full LMS, a production-scale proctoring system, or a public deployment during the main project window. Deployment is planned as a follow-up in late July.

\section{AI Usage}

\subsection{AI Tooling Inventory}

AI was used both as a product feature and as a development aid. The product itself integrates DeepSeek through a backend AI completion service for student assistance, administrator draft generation, and AI-usage grading. During development, team members used Codex and Claude for coding support, planning, debugging, documentation, and review.

We also created an agent-oriented development harness inside the repository. The active \texttt{AGENTS.md} file acts as a central orchestration guide. It defines the mission, module boundaries, security rules, state-machine-first discipline, testing expectations, and review procedure. The \texttt{.agents/skills/} folder contains reusable role checklists such as module routing, frontend IDE work, backend assessment work, sandbox execution work, AI telemetry work, full-stack integration, strict code review, and handoff summaries.

The repository also contains \texttt{mcp-code-analyzer}, a lightweight MCP server used as an advisory requirements-compliance scanner. Its purpose was not to prove correctness automatically, but to give agents and developers another structured review signal against \texttt{SPEC.md} and project heuristics.

\subsection{AI in Coding}

AI helped most when the task was already well bounded. Examples include drafting endpoint tests, comparing frontend API expectations with backend contracts, explaining failing startup paths, finding likely integration risks, and producing first-pass documentation. Codex was particularly useful for repository-wide inspection because it could read the codebase, run checks, and keep the current module boundary in mind.

We did not treat generated code as finished work. The practical workflow was:

\begin{enumerate}
  \item describe the task against \texttt{SPEC.md} and the relevant module,
  \item let the agent inspect local files before editing,
  \item make a small implementation change,
  \item run targeted tests or type checks,
  \item review the diff manually,
  \item update documentation when behavior or contracts changed.
\end{enumerate}

This was especially important for AI and sandbox code. A plausible-looking AI response is not enough for this project, because hidden tests, provider calls, token accounting, and attempt ownership are security-sensitive.

\subsection{AI in Individual Planning}

Team members used AI differently depending on their role:

\begin{itemize}
  \item The frontend work used AI to break down pages, state flows, loading/error states, and API adapter changes.
  \item The backend work used AI to reason about endpoint contracts, EF Core persistence, session ownership, and report aggregation.
  \item QA used AI to turn requirements into checklists and to identify regression tests around startup, scoring, and session behavior.
  \item UI/UX work used AI to explore workflow diagrams and simplify complex technical flows into readable visuals.
  \item PM and AI engineering used AI to refine scope, organize module boundaries, design the agent harness, and document how AI should be used safely.
\end{itemize}

\subsection{AI in Team Planning}

The strongest improvement came after we made the AI workflow itself explicit. Instead of asking an assistant to ``fix the project,'' we routed work through documented modules. The agent was expected to identify the goal, context, constraints, and done condition; choose the right module; keep unrelated modules untouched; and finish with review.

This mattered because the project had four major modules: identity/assessment management, frontend workspace, sandboxed execution, and AI telemetry/assistance. Once these boundaries were written down, AI help became less random. Prompts became more specific, generated plans were easier to challenge, and hallucinated changes were easier to reject.

\subsection{Prompt Engineering Strategies}

Our most effective prompt strategy was not a single clever prompt. It was context management. We gave the AI:

\begin{itemize}
  \item the active specification priority,
  \item the relevant module boundary,
  \item the acceptance criteria,
  \item the files it should inspect,
  \item the tests it should run,
  \item the rule that hidden tests, AI keys, and session ownership must remain backend-side.
\end{itemize}

The local skill chain also helped. A task could start with routing, move to one primary coder skill, use companion skills only for boundary checks, and finish with strict review. This reduced the tendency for AI to over-edit or solve a frontend issue by changing backend behavior without a reason.

\subsection{Lessons Learned}

The biggest lesson was that AI speeds up exploration more than it removes engineering work. Early in the project, while parts of the system still used mock objects and placeholder behavior, AI sometimes hallucinated around the boundary between mock and real artifacts. This became worse when the codebase and documentation disagreed. After we spent a long night updating the docs, aligning the codebase, and strengthening the agent instructions, the situation improved significantly.

We also learned that verification cost is real. AI can produce a working-looking endpoint, UI state, or prompt quickly, but the team still has to check whether it respects auth, persistence, hidden-test boundaries, and failure behavior. Our practice changed from ``generate then inspect'' to ``define constraints, generate a small change, test, then review.'' That slower loop produced better results.

For presentation material and this report, GenAI was also used, but not blindly. AI helped draft and organize language, while the team provided the project details, corrected tone, selected diagrams, and planned manual review before submission.
