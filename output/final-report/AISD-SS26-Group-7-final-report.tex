\subsection{Purpose of the Platform}

We built the platform as a \textbf{browser-based coding assessment system for practical software development tasks}. Figure~\ref{fig:user-guide-flow} shows what each role does. Administrators prepare and publish assessments, manage accounts, and review results. Students solve the tasks in an IDE-like workspace, use the built-in AI assistant when it is available, submit their work, and briefly explain how they used the AI.

For now, the project is intended for local demonstrations and course evaluation. It will not be hosted until we have combined the User Acceptance Testing (UAT) findings with the course feedback, made the resulting changes, and tested the revised version. The instructions below therefore describe the local setup.

\subsection{Local Setup}

Getting the project running locally takes one main command. Before starting, install Node.js 20 or newer, npm, and a .NET SDK that is compatible with the backend solution. A Docker-compatible runtime is optional for browsing the application, but required for running code and grading final submissions.

\begin{enumerate}
  \item Clone the repository:
  \begin{verbatim}
git clone https://github.com/ZAnimenl/AISD-SS26-Group-7.git
cd AISD-SS26-Group-7
  \end{verbatim}
  \item Create or edit \texttt{.env.local} in the repository root before starting the application. If necessary, copy \texttt{.env.example} first. Then add the AI provider settings:
  \begin{verbatim}
Deepseek__ApiKey=<your-deepseek-api-key>
Deepseek__Enabled=true
  \end{verbatim}
  Keep the real API key only in \texttt{.env.local}; never commit it to the repository.
  \item Start the full local application:
  \begin{verbatim}
npm run dev
  \end{verbatim}
  \item Open the frontend at:
  \begin{verbatim}
http://localhost:3000
  \end{verbatim}
  \item The backend API runs locally at:
  \begin{verbatim}
http://localhost:5140/api/v1
  \end{verbatim}
\end{enumerate}

The startup script handles the rest: it prepares the SQLite database in \texttt{.local-data/}, loads \texttt{.env.local}, restores missing dependencies, starts the backend, waits for its health check, and then launches the Next.js frontend. The application still opens when Docker is unavailable. In that case, the workspace clearly disables final submission until it can reach the sandbox runtime.

\subsection{Signing In}

For a local demo, click the \textbf{development-only ``Use local admin'' button} on the login page. It fills in these default credentials:

\begin{verbatim}
admin@example.com
Admin123!
\end{verbatim}

The backend creates or repairs this account during startup. Once signed in, the local administrator can create additional administrator accounts from the role-management page. There is no public administrator registration, which prevents users from assigning themselves elevated access.

\subsubsection{Creating and Managing Assessments}

Administrators create, edit, and remove assignments from the assessment section. Each assessment has a title, description, duration, availability window, publication status, AI setting, and one or more practical questions. Four task categories are currently supported:

\begin{itemize}
  \item frontend UI extension,
  \item REST API development,
  \item database query/schema work,
  \item bug fixing in existing code.
\end{itemize}

Assessment creation has three steps: define the basics, prepare the questions, and confirm the delivery settings. AI can produce a first draft, but it never publishes the task. An administrator must still check the wording, starter files, visible and hidden tests, metadata, and scoring setup. Video~\ref{vid:admin-creation} shows the full process.

\subsubsection{Reports and Scoring}

Reports deliberately separate \textbf{functional correctness} from \textbf{AI usage}. This lets an administrator discuss the quality of the submitted code without mixing it with the way the student used AI. When AI is enabled, both scores contribute to the final result; otherwise, the functional score stands on its own.

An administrator can start with an assessment summary and then open the evidence for an individual student. That evidence includes prompts and responses, interaction types, token use, per-task metrics, decisions on suggestions, the final reflection, grading status, confidence, and any administrator retry status.

\subsection{Student Workflow}

\subsubsection{Registration and Login}

Students can sign in with email and password or, when configured, with Google OAuth. Email registration uses a \textbf{verification code}: the student enters the account details, receives the code, and confirms it to finish registration. Local development can expose a fallback code so an SMTP problem does not block a demonstration.

\subsubsection{Dashboard and Assessment List}

After signing in, students see which assessments are available, in progress, or complete. The list also shows the duration, availability dates, number of questions, and whether AI assistance is enabled.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=0.85\linewidth]{\figpath/demo-student-dashboard.png}
  \caption{Student dashboard showing available assessments, recent activity, and score preview.}
\end{figure}

\subsubsection{Starting an Assessment}

The start page gives students one final overview before the timer begins: title, description, duration, availability window, question count, and AI setting. Starting from this page creates the attempt. Video~\ref{vid:student-assessment} follows the rest of the journey through the workspace, submission, and reflection.

\subsubsection{Assessment Workspace}

The workspace should feel familiar to anyone who has used an IDE. Its compact layout is divided into five areas:

\begin{itemize}
  \item The left side contains the task panel and problem statement.
  \item The center contains a Monaco-based editor with multi-file support.
  \item File controls allow students to switch files, create additional allowed files, and select a supported language.
  \item The right side contains the AI assistant and AI usage display.
  \item The bottom output panel contains preview and console tabs for run feedback.
\end{itemize}

Code is never executed in the browser. Both Run and Submit call assessment-scoped backend APIs, where the active attempt is resolved from the signed-in user and assessment ID. If the backend cannot reach Docker, final submission is disabled and the workspace explains why; it does not show a successful grade that was never produced.

\embeddedvideo{User_Assessment.mp4}{\figpath/video-user-assessment.jpg}{Student assessment workflow from opening an assessment through workspace activity, submission, and reflection.\label{vid:student-assessment}}

\subsubsection{AI Assistance and Reflection}

When an assessment allows AI, students can ask for suggestions, explanations, or debugging help. The assistant sees the visible task, workspace files, selected language, active file, and public run output. Hidden tests, grading code, provider keys, and server-only prompts stay on the backend.

Submission is followed by a reflection of no more than 100 words. Students describe how they used the assistant, name one useful suggestion and how they checked it, and note something they rejected, corrected, or found unhelpful. A backend-controlled timer keeps this step consistent, and the response becomes part of the AI-usage grade.

\subsubsection{Results}

Once submission and any AI-usage grading are complete, students can view their functional score, AI-usage score when applicable, final score, reflection status, and feedback summary. The administrator report adds the underlying evidence and controls for retrying a failed grade.

\section{Technical Documentation}

\subsection{System Overview}

Coding education now faces a mismatch: students use AI in everyday development, while many assessments still look only at the final code. Our platform records both sides of the work. It grades the submitted program and keeps an auditable history of how the built-in assistant was used. The aim is not to penalize AI use, but to assess it as part of a realistic development process.

The working MVP covers:

\begin{itemize}
  \item administrator-managed assessments, questions, test cases, and users,
  \item student registration, login, dashboard, assessment start, workspace, reflection, and results,
  \item backend-owned attempts and workspace persistence,
  \item Docker-based run/submit architecture for public and hidden tests,
  \item embedded AI assistance through a backend AI layer,
  \item AI interaction logging, token tracking, reflection, and automatic AI usage scoring,
  \item reporting for functional, AI-usage, and final scores.
\end{itemize}

\subsection{Overall Architecture}

Figure~\ref{fig:arch-clean} gives the high-level architecture. Students and administrators use the same Next.js web application, which calls an ASP.NET Core API. The backend is the system of record for authentication, assessments, workspace state, AI integration, grading, reports, and database access. The browser never connects directly to an external service.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\linewidth]{\figpath/architecture-clean.png}
  \caption{Clean system architecture.}
  \label{fig:arch-clean}
\end{figure}

Figure~\ref{fig:arch-detailed} opens up those layers. On the frontend, student and administrator pages share a central API client with the coding workspace. The backend separates authentication and roles, assessments, attempts, workspace persistence, execution and grading, reports, and AI. EF Core handles persistent data, Docker isolates code execution, and all DeepSeek requests pass through the backend AI service.

\subsection{AI Architecture}

The AI layer has three jobs: help students in the workspace, prepare draft tasks for administrators, and grade AI usage after submission from the recorded conversation, token data, execution history, and reflection.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\linewidth]{\figpath/ai-workflow-clean.png}
  \caption{Clean AI workflow across student assistance, admin generation, and grading.}
  \label{fig:ai-clean}
\end{figure}

Figure~\ref{fig:ai-clean} shows the common path behind all three jobs. Every request first reaches a backend endpoint, where the user and role are checked and a safe context is assembled. The backend then selects the right workflow, requests a structured completion when needed, and stores the resulting transcript, tokens, events, score, or draft. The colored bands make the boundary clear: student and administrator requests differ at the edges, but both rely on the same backend-controlled provider and persistence layers.

\begin{figure}[H]
  \centering
  \includegraphics[angle=90,height=0.76\textheight]{\figpath/ai-workflow-student-detailed.png}
  \caption{Detailed student AI assistance workflow.}
  \label{fig:ai-student}
\end{figure}

Figure~\ref{fig:ai-student} follows a student request in detail. The browser sends the selected language, active and visible files, message, and latest public run result. It does not decide which attempt owns the request; the backend derives that from the authenticated student and assessment. Hidden tests, grading logic, provider keys, and server-only prompts are removed before the provider call. Only the markdown response and safe workspace actions return to the browser, while the interaction and token use are stored for later review.

\begin{figure}[H]
  \centering
  \includegraphics[angle=90,height=0.76\textheight]{\figpath/ai-workflow-admin-detailed.png}
  \caption{Detailed administrator AI draft-generation and AI-usage grading workflow.}
  \label{fig:ai-admin}
\end{figure}

Figure~\ref{fig:ai-admin} follows the two administrator paths. Draft generation starts with a role check and a constrained prompt, then stores the returned questions, starter files, tests, and metadata for human review. AI-usage grading instead loads the student's submission, transcript, token use, and reflection. It may call the provider before saving the score, summary, confidence, and supporting details.

\subsection{Workflow Implementation and Trade-offs}

We deliberately kept every AI call behind the backend. A direct browser-to-provider connection would have been simpler, but it would also have exposed more state and reduced our control over roles, prompts, logging, token accounting, and failures.

This choice gave us several practical benefits:

\begin{itemize}
  \item hidden tests and grading details never enter the student UI,
  \item AI provider keys stay server-side,
  \item every AI interaction can be logged with token usage and semantic tags,
  \item administrator review remains part of generated task publication,
  \item AI usage grading has access to a full attempt history rather than a single prompt.
\end{itemize}

It also created extra work:

\begin{itemize}
  \item more backend contract work is needed for every frontend AI feature,
  \item provider errors must be represented clearly to users,
  \item generating good tasks requires strict validation and correction loops,
  \item local demonstrations depend on real external configuration for AI and Docker.
\end{itemize}

\section{Project Management}

\subsection{Project Vision}

We started with a simple question: how can a coding assessment acknowledge AI without losing sight of the student's own work? Our answer was a browser-based environment where students solve realistic tasks with a recorded assistant, while administrators can review both code correctness and AI use. The same platform also had to support task creation, draft review, user management, and reporting.

From that vision, we set five concrete deliverables:

\begin{itemize}
  \item a working Next.js frontend with student and administrator workflows,
  \item an ASP.NET Core backend with authentication, RBAC, assessments, attempts, workspace state, reports, and AI endpoints,
  \item a sandboxed grading architecture based on Docker,
  \item AI assistance, task generation, transcript logging, token tracking, and AI usage scoring,
  \item documentation, diagrams, startup tooling, and verification tests.
\end{itemize}

We also agreed on what not to build. A full learning-management system, production-scale proctoring, and public hosting were outside the project window. The next stage will be driven by evidence: we will combine the UAT findings with the formal course feedback, address the most important issues, test the revision, and deploy only when that version is ready.

\section{AI Usage}

\subsection{AI Tooling Inventory}

We used AI in two distinct ways. Inside the product, DeepSeek powers student assistance, administrator draft generation, and AI-usage grading through a backend completion service. During development, the team used Codex and Claude for planning, coding support, debugging, documentation, and review.

To make that support more consistent, we added an agent-oriented development harness to the repository. Its \texttt{AGENTS.md} file sets out the project goal, module boundaries, security rules, state-machine discipline, testing expectations, and review process. Reusable checklists in \path{.agents/skills/} cover routing, frontend IDE work, backend assessments, sandbox execution, AI telemetry, full-stack integration, code review, and handover.

The repository also includes \texttt{mcp-code-analyzer}, a lightweight MCP server that checks changes against \texttt{SPEC.md} and a set of project heuristics. We treated its output as a review aid, not as proof that the code was correct.

\subsection{AI in Coding}

AI was most useful when we had already defined the problem. It helped draft endpoint tests, compare frontend expectations with backend contracts, trace failed startup paths, flag integration risks, and prepare first versions of documentation. Codex was especially helpful for repository-wide checks because it could inspect the code, run commands, and follow the module boundaries at the same time.

Generated code was never considered finished on arrival. Figure~\ref{fig:ai-development-orchestration} shows the loop we used to constrain a task, route it to the right module, verify the result, and decide whether to accept it.

\begin{figure}[H]

That discipline mattered most around AI and sandbox code. Something that merely looks plausible can still leak a hidden test, miscount tokens, call the wrong provider path, or attach work to the wrong attempt.

\subsection{AI in Individual Planning}

Each role used AI for a different kind of problem:

\begin{itemize}
  \item Frontend work used it to break pages into state flows, loading and error cases, and API adapter changes.
  \item Backend work used it to reason through endpoint contracts, EF Core persistence, attempt ownership, and report aggregation.
  \item QA turned requirements into checklists and identified regression cases around startup, scoring, and session behavior.
  \item UI/UX work explored workflow diagrams and translated dense technical processes into readable visuals.
  \item Project management and AI engineering used it to refine scope, organize module boundaries, design the agent harness, and document safe use.
\end{itemize}

\subsection{AI in Team Planning}

Our results improved once we stopped giving broad requests such as ``fix the project.'' Every AI-assisted task had to state the goal, relevant context, constraints, and definition of done. It was then routed to one documented module, with unrelated areas left alone, and ended with a review.

The project has four major areas: identity and assessment management, the frontend workspace, sandboxed execution, and AI telemetry and assistance. Writing those boundaries down made the output less unpredictable. Prompts became more specific, plans were easier to question, and irrelevant or invented changes stood out sooner.

\subsection{Prompt Engineering Strategies}

The most effective prompt technique was not clever wording; it was careful context. For each task, we supplied:

\begin{itemize}
  \item the active specification priority,
  \item the relevant module boundary,
  \item the acceptance criteria,
  \item the files it should inspect,
  \item the tests it should run,
  \item the rule that hidden tests, AI keys, and session ownership must remain backend-side.
\end{itemize}

The local skill chain reinforced the same habit. A task started with routing, used one primary coding skill, called companion skills only for boundary checks, and finished with a strict review. This reduced over-editing and made it less likely that a frontend problem would be ``solved'' by changing backend behavior without justification.

\subsection{Lessons Learned}

Our clearest lesson was that AI speeds up exploration; it does not remove the engineering work. Early on, some modules still mixed mock objects with real behavior, and the AI occasionally invented connections between them. Out-of-date documentation made this worse. After a long evening spent aligning the code and documentation and tightening the agent instructions, the suggestions became noticeably more reliable.

Verification still took real time. An endpoint, UI state, or prompt could look complete while violating authentication, persistence, hidden-test boundaries, or failure handling. We therefore moved from ``generate, then inspect'' to a smaller loop: set the constraints, make one limited change, run the relevant checks, and review it. The loop felt slower but produced far fewer surprises.

We used GenAI for presentation material and early report drafts as well. The team supplied the evidence, corrected the tone and technical details, chose the diagrams, and reviewed the final material before submission.

\section{User Acceptance Testing}

\subsection{Case Study and Method}

We ran the UAT in person with \textbf{15 students}. Every participant had passed \textbf{INHN0001} (Introduction to Informatics), where the coding assessments did not allow AI assistance. Their average self-rated programming experience was \textbf{3.8 out of 5}. All results in this report are anonymous.

This participant group was selected deliberately. Having completed a conventional coding assessment, the students already knew what it meant to solve a programming task and demonstrate their own ability under exam conditions. They had also experienced the restriction of working without AI, even though AI-assisted development is becoming common outside the classroom. This made them a useful group for judging whether recorded, limited AI support could make an assessment feel more realistic without hiding individual competence or weakening fairness.

Each participant opened an assessment, edited and ran code, used the assistant, submitted the work, and completed the reflection. They then answered a nine-question survey. All 15 finished the survey, and \textbf{eight suggested future improvements}. We did not record task duration, device type, moderator interventions, or run-check history, so the results should be read as evidence from a pilot rather than a controlled study or production-readiness test.

\subsection{Results}

All 15 students reached the end of the workflow: \textbf{10 without help} and \textbf{five with minor help}. Ease of use averaged \textbf{4.53 out of 5}, AI usefulness \textbf{4.0 out of 5}, and the overall platform rating \textbf{4.67 out of 5}. The averages do not tell the whole story. Eight students encountered an important problem, and the AI usefulness responses included three ratings of 2 and two ratings of 3. Everyone supported another classroom test, although \textbf{eight wanted minor changes and four wanted major changes beforehand}.

These recommendations had not yet been implemented at the report cutoff. AI reliability is the most urgent issue: the positive average masks several cases where repeated assistance still did not help the code pass its run checks. Visual polish was the other recurring concern, followed by clearer onboarding and faster loading. We will combine these findings with the formal course feedback when planning the revision that precedes deployment.

We therefore interpret the outcome as \textbf{conditional acceptance for further classroom testing}. All participants wanted testing to continue, but 12 of 15 asked for changes first. This pilot cannot establish production or enterprise readiness. A larger follow-up should record task time, run-check outcomes, AI retries, error recovery, and moderator help so that improvements can be measured rather than inferred from ratings alone.
