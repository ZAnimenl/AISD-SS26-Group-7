# SPEC.md

## Note

This version expands the original specification with structured AI assistance, AI credit budgets, AI Rescue, LLM-based task generation, post-submission reflection, process-aware scoring, and controlled student report release.

Requirement IDs in this version are the authoritative IDs for future implementation prompts, tests, and review notes. The document intentionally uses both numbered IDs such as `REQ-01` and feature-scoped IDs such as `REQ-HINT-01`, `REQ-CREDIT-01`, `REQ-RESCUE-01`, `REQ-REFLECT-01`, `REQ-TASKGEN-01`, and `REQ-SCORE-01`.

This project does not require separate `design.md` or `tasks.md` files. Implementation prompts, code changes, tests, and review notes should reference relevant requirement IDs where practical.

---

## 1. Overview

### 1.1 Project

Online AI-Coding Assessment / Interview Platform

### 1.2 Problem Statement

Traditional coding assessments mainly evaluate the final code output. In an AI-assisted development setting, this is no longer sufficient. The platform must support realistic AI-assisted coding workflows while still allowing administrators to assess candidate performance, code quality, AI usage behavior, and the candidate's ability to critically evaluate AI-generated code.

### 1.3 Objective

Build a web-based platform that lets administrators create and manage coding assessments, lets students complete them in a browser-based IDE-like environment with structured AI support, executes code safely in a sandbox, and produces reports that summarize assessment outcomes, AI interaction patterns, AI credit usage, AI Rescue behavior, student reflection, and LLM-based process evaluation.

### 1.4 Scope

This document defines the requirements for the system in a structured, testable form suitable for spec-driven development. It focuses on stakeholder needs, user stories, acceptance criteria, constraints, and verifiable requirement statements.

## 2. Terminology

- **Normal AI assistance** means assessment-time help requested through configured structured hint levels.
- **Structured hint level** means a predefined assistance category such as concept hint, strategy hint, debugging hint, pseudocode hint, or code-level suggestion.
- **Unrestricted free-form AI chat** means an open-ended chat interface where a student can ask arbitrary questions without selecting a configured structured assistance mode. This is not allowed during assessments.
- **AI credit** means a finite per-question budget used for normal AI assistance.
- **AI Rescue** means a separate emergency assistance feature that provides strong code-level suggestions and is not charged against normal AI credits.
- **Correct AI Rescue suggestion** means a harmless code-level suggestion that is intended by the system or LLM to solve the relevant question correctly.
- **Misleading AI Rescue suggestion** means a harmless, normal-looking code-level suggestion that is intentionally logically wrong for the relevant question. It must not contain malicious code, platform-breaking code, security-risk code, hidden test details, or instructions that bypass platform rules.
- **Rescue correctness label** means the system-side label indicating whether a rescue suggestion was generated as correct or misleading. Students must not see this label during the assessment.
- **Student rescue decision** means whether the student accepts, rejects, or modifies an AI Rescue suggestion.
- **Post-submission reflection** means a short timed form shown after the student clicks submit and before the attempt is marked fully complete, when the feature is enabled.
- **LLM-based process evaluation** means a decision-support evaluation generated by an LLM from submitted code, AI usage logs, rescue behavior, and reflection text.
- **Process-aware score** means an administrator-facing score that combines code correctness and process indicators. It supports administrator decisions and must not be treated as a fully autonomous final grading or hiring decision.
- **Released student report** means the final result view that students can see only after an administrator releases results.

## 3. Goals

- Provide a realistic AI-assisted coding assessment environment.
- Support browser-based coding for at least Python and JavaScript/TypeScript.
- Allow administrators to create and manage assessments and questions.
- Allow administrators to generate draft coding tasks with an LLM and review them before publication.
- Automatically evaluate submitted code using predefined test cases.
- Provide structured AI assistance through hint levels instead of unrestricted free-form AI chat.
- Limit normal AI assistance through a per-question AI credit budget based on task difficulty.
- Provide an AI Rescue feature that tests whether students can critically judge AI-generated code.
- Record and summarize AI interactions during assessments.
- Evaluate code correctness and AI-assisted problem-solving behavior, including hint usage, rescue behavior, reflection quality, and critical evaluation of AI-generated code.
- Generate administrator-facing reports about performance, AI usage, AI credits, rescue behavior, reflection, LLM-based process evaluation, and process-aware scoring.
- Support a workflow where implementation prompts and future code changes can be traced back to requirement IDs in this specification.

---

## 4. Non-Goals

- Real-time collaborative editing between multiple students.
- Full proctoring features such as webcam monitoring.
- Production-grade plagiarism detection.
- Support for all programming languages.
- A fully autonomous hiring decision system.
- Fine-grained code authorship attribution at token or line level.
- Fully reliable automated judgment of student intent.
- Student access to rescue correctness labels during an assessment.
- Unrestricted free-form AI chat during an assessment.

---

## 5. Stakeholders

- **Students / Candidates**: complete coding assessments and use structured AI assistance where enabled.
- **Administrators / Instructors / Interviewers**: create assessments, generate draft tasks, review results, analyze AI usage, release student reports, and review process evaluation reports.
- **Developers / Maintainers**: implement and maintain the platform.
- **Course Staff / Hiring Teams**: consume results for grading or screening.

## 6. User Stories

### US-01 Assessment Creation

As an administrator, I want to create and manage coding assessments so that I can evaluate students or candidates in a structured way.

### US-02 Online Coding

As a student, I want to solve coding tasks in a browser-based IDE so that I do not need to configure a local environment during the assessment.

### US-03 Safe Code Execution

As a platform operator, I want student code to run in an isolated sandbox so that the system remains secure and stable.

### US-04 Multi-Language Support

As a student, I want to solve tasks in supported languages such as Python or JavaScript/TypeScript so that the platform matches common assessment needs.

### US-05 Structured AI-Assisted Problem Solving

As a student, I want access to structured AI assistance during the assessment so that I can receive controlled help without using unrestricted AI chat.

### US-06 Automated Evaluation

As an administrator, I want submissions to be evaluated automatically using test cases so that scoring is fast, consistent, and reproducible. The evaluation relies on test code defined by the administrator.

### US-07 AI Usage Review

As an administrator, I want to review how AI was used during an assessment so that I can judge process as well as final output.

### US-08 Reporting

As an administrator, I want performance reports and AI usage summaries so that I can compare users and review outcomes efficiently.

### US-09 LLM-Based Task Generation

As an administrator, I want to generate draft coding tasks with an LLM so that I can create assessment content more efficiently while still reviewing it before publication.

### US-10 AI Credit Budget

As an administrator, I want normal AI assistance to use a limited credit budget so that students must use AI help strategically.

### US-11 AI Rescue and Critical Evaluation

As an administrator, I want students to receive limited emergency AI Rescue suggestions, including intentionally imperfect suggestions, so that I can evaluate whether students blindly trust or critically judge AI-generated code.

### US-12 Post-Submission Reflection

As an administrator, I want students to explain their solution and AI usage after submission so that I can assess whether they understand their submitted work.

### US-13 Controlled Student Result Release

As an administrator, I want to control when students can view final scores, LLM evaluations, and short explanations so that results can be reviewed before release.

## 7. Functional Requirements

### 7.1 Authentication and Access Control

- **REQ-01** The system shall support authenticated access for at least two roles: student and administrator.
- **REQ-02** When a user attempts to access a protected page without authentication, the system shall require login before access is granted.
- **REQ-03** When a student is authenticated, the system shall only allow access to student-relevant features.
- **REQ-04** When an administrator is authenticated, the system shall allow access to assessment management, submission review, report release, and reporting features.
- **REQ-05** If a user attempts to access a feature outside their role permissions, then the system shall deny access.

### 7.2 Assessment Management

- **REQ-06** The system shall allow an administrator to create an assessment with a title, description, duration, and status.
- **REQ-07** The system shall allow an administrator to edit an existing assessment.
- **REQ-08** The system shall allow an administrator to archive or delete an assessment.
- **REQ-09** When an administrator adds questions to an assessment, the system shall associate those questions with that assessment.
- **REQ-10** The system shall allow an administrator to set whether an assessment is draft, active, closed, or archived.
- **REQ-11** The system shall allow an administrator to enable or disable AI assistance for an assessment.
- **REQ-12** Where AI assistance is enabled, the system shall allow an administrator to configure whether AI hint levels, AI credits, AI Rescue, and post-submission reflection are enabled for the assessment.
- **REQ-13** The system shall allow an administrator to configure the AI Rescue correctness probability at assessment level.
- **REQ-14** The default AI Rescue correctness probability shall be 50% correct and 50% misleading.
- **REQ-15** The system shall allow an administrator to control whether final student reports are released for an assessment.

### 7.3 Question Management

- **REQ-16** The system shall allow an administrator to create a coding question.
- **REQ-17** The system shall store for each question at least: title, problem description, supported language or languages, difficulty, and grading configuration.
- **REQ-18** Where starter code is provided, the system shall present that starter code in the coding workspace.
- **REQ-19** The system shall allow an administrator to define one or more test cases for each question.
- **REQ-20** The system shall allow an administrator to edit or remove a question before or after association with an assessment.

### 7.4 LLM-Based Task Generation

- **REQ-TASKGEN-01** The system shall allow administrators to generate draft coding questions using an LLM.
- **REQ-TASKGEN-02** When generating a task, the administrator shall be able to specify at least topic, language, difficulty, expected duration, and desired number of test cases.
- **REQ-TASKGEN-03** The generated task shall include at least a title, problem statement, starter code, test cases, and solution outline.
- **REQ-TASKGEN-04** Generated tasks shall be saved as drafts before publication.
- **REQ-TASKGEN-05** The administrator shall be able to review and edit generated tasks before adding them to an active assessment.
- **REQ-TASKGEN-06** The administrator shall be able to accept, edit, reject, regenerate, or add a generated task to an assessment.
- **REQ-TASKGEN-07** Generated test cases shall remain draft content until an administrator reviews them and marks them as public or hidden.
- **REQ-TASKGEN-08** If LLM task generation fails, then the system shall show an actionable error message and shall not publish incomplete generated content automatically.

### 7.5 Student Assessment Participation

- **REQ-21** When a student opens an available assessment, the system shall display the assessment title, description, time information, and available questions.
- **REQ-22** The system shall allow a student to open a question and view its problem statement in the browser.
- **REQ-23** The system shall allow a student to write and edit code in the browser during an assessment attempt.
- **REQ-24** The system shall allow a student to run code before final submission.
- **REQ-25** When a student submits a solution, the system shall store the submission and trigger evaluation.
- **REQ-26** If an assessment is closed or expired, then the system shall prevent new submissions.
- **REQ-27** If final reports have not been released by an administrator, then the system shall prevent students from viewing final scores, LLM evaluations, and final report explanations for that assessment.
- **REQ-28** If final reports have been released by an administrator, then the system shall allow students to view their own final scores, LLM evaluations, and short explanations for that assessment.

### 7.6 IDE-Like Workspace

- **REQ-29** The system shall provide a browser-based code editor.
- **REQ-30** The editor shall support syntax highlighting for all supported languages.
- **REQ-31** The workspace shall provide an output or console area for program output and runtime errors.
- **REQ-32** The workspace shall provide a visible action for code execution.
- **REQ-33** While an assessment attempt is active, the system shall preserve the student's in-progress code during normal page interactions.

### 7.7 Sandboxed Code Execution

- **REQ-34** The system shall execute user code in an isolated sandboxed environment.
- **REQ-35** If submitted code exceeds configured time or resource limits, then the system shall terminate execution and report the failure.
- **REQ-36** The system shall prevent user code from directly accessing sensitive server resources.
- **REQ-37** When code execution completes, the system shall return execution status, standard output, and runtime error information to the user when appropriate.

### 7.8 Language Support

- **REQ-38** The system shall support Python for code editing and execution.
- **REQ-39** The system shall support JavaScript and/or TypeScript for code editing and execution.
- **REQ-40** Where a question restricts allowed languages, the system shall enforce those restrictions during submission.
- **REQ-41** The first student-facing implementation shall support Python and JavaScript as student submission languages.

### 7.9 Structured AI Assistance and Hint Levels

- **REQ-HINT-01** The system shall provide structured AI assistance during assessments where AI assistance is enabled.
- **REQ-HINT-02** Where structured AI assistance is enabled, the system shall support AI hint levels, including at least concept hint, strategy hint, debugging hint, pseudocode hint, and code-level suggestion.
- **REQ-HINT-03** During assessments, the system shall not provide unrestricted free-form AI chat.
- **REQ-HINT-04** During assessments, AI assistance shall be provided only through configured structured assistance modes, such as hint levels or AI Rescue.
- **REQ-HINT-05** When a student requests normal AI assistance, the system shall require the student to select a hint level before generating the AI response.
- **REQ-HINT-06** When a student uses AI assistance, the system shall log the interaction with the associated user, assessment, question, selected hint level, credit cost, prompt or request text if applicable, AI response, and timestamp.
- **REQ-HINT-07** If AI assistance is disabled for a given assessment, then the system shall prevent access to AI assistance features for that assessment.

### 7.10 AI Credit Budget

- **REQ-CREDIT-01** Where normal AI assistance is enabled, the system shall assign an AI credit budget per question based on question difficulty.
- **REQ-CREDIT-02** The default per-question AI credit budget shall be 6 credits for easy questions, 10 credits for medium questions, and 15 credits for hard questions.
- **REQ-CREDIT-03** When a student requests normal AI assistance, the system shall deduct credits from that question's remaining AI credit balance according to the selected hint level.
- **REQ-CREDIT-04** If the student has insufficient credits for the selected hint level, then the system shall prevent the request and display the remaining credit balance.
- **REQ-CREDIT-05** The administrator shall be able to override the default AI credit budget for a question or assessment.
- **REQ-CREDIT-06** The system shall display the remaining AI credits for the active question during the assessment.
- **REQ-CREDIT-07** The system shall record AI credit usage for administrator reporting.
- **REQ-CREDIT-08** The default normal hint costs shall be: concept hint 1 credit, strategy hint 2 credits, debugging hint 3 credits, pseudocode hint 4 credits, and code-level suggestion 6 credits.

### 7.11 AI Rescue

- **REQ-RESCUE-01** Where AI Rescue is enabled, the system shall provide each student with 4 AI Rescue chances per assessment.
- **REQ-RESCUE-02** AI Rescue chances shall be shared across all questions in the assessment.
- **REQ-RESCUE-03** The AI Rescue feature shall be independent from normal AI hints and AI credits.
- **REQ-RESCUE-04** When a student uses the AI Rescue feature, the system shall provide a strong code-level AI suggestion.
- **REQ-RESCUE-05** For each AI Rescue request, the system shall randomly generate either a correct rescue suggestion or a misleading rescue suggestion according to the assessment-level correctness probability.
- **REQ-RESCUE-06** The default AI Rescue correctness probability shall be 50% correct and 50% misleading.
- **REQ-RESCUE-07** Administrators shall be able to edit the AI Rescue correctness probability at assessment level.
- **REQ-RESCUE-08** Misleading rescue suggestions shall be harmless, normal-looking, logically wrong solutions for the current question.
- **REQ-RESCUE-09** Misleading rescue suggestions shall not contain malicious code, platform-breaking code, security-risk code, hidden test details, or instructions that bypass platform rules.
- **REQ-RESCUE-10** When an AI Rescue suggestion is shown, the student shall choose whether to accept, reject, or modify the suggestion.
- **REQ-RESCUE-11** The system shall record whether each rescue suggestion was generated as correct or misleading.
- **REQ-RESCUE-12** The system shall record whether the student accepted, rejected, or modified each rescue suggestion.
- **REQ-RESCUE-13** The system shall record how many AI Rescue chances remain for the student during the assessment.
- **REQ-RESCUE-14** Rescue correctness labels shall be hidden from students during the assessment.
- **REQ-RESCUE-15** When an AI Rescue suggestion is shown, the system shall start a response timer and record how long the student takes before accepting, rejecting, or modifying the suggestion.
- **REQ-RESCUE-16** If a student accepts a rescue suggestion within 3 seconds, the system shall record this as a possible low-critical-review indicator, even if the accepted suggestion is correct.
- **REQ-RESCUE-17** If a student correctly rejects a misleading AI Rescue suggestion, the system shall record this as a positive critical AI judgment indicator.
- **REQ-RESCUE-18** If a student modifies an imperfect or misleading rescue suggestion in a way that improves correctness, the system shall record this as a positive critical AI judgment indicator.
- **REQ-RESCUE-19** The LLM-based process evaluation shall consider whether the student accepted correct rescue suggestions, rejected misleading rescue suggestions, modified imperfect suggestions appropriately, and spent reasonable time reviewing rescue output before deciding.
- **REQ-RESCUE-20** The administrator report shall include AI Rescue usage and critical evaluation behavior.

### 7.12 Automated Evaluation

- **REQ-42** When a solution is submitted, the system shall evaluate it against predefined test cases.
- **REQ-43** The system shall determine whether each relevant test case passes or fails.
- **REQ-44** The system shall calculate and store an overall result, score, or pass/fail outcome for the submission.
- **REQ-45** Where execution fails before all test cases run, the system shall store the failure status.

### 7.13 Submission, Reflection, and Result Storage

- **REQ-46** The system shall store each submission with its source code, selected language, timestamp, and related identifiers.
- **REQ-47** The system shall store evaluation results linked to the corresponding submission.
- **REQ-48** Where multiple submissions are allowed, the system shall preserve submission history.
- **REQ-REFLECT-01** When a student clicks the submit button and post-submission reflection is enabled, the system shall display a pop-up reflection form before the assessment attempt is marked fully complete.
- **REQ-REFLECT-02** After the student clicks submit, the system shall freeze the submitted code for grading, even while the reflection form is still open.
- **REQ-REFLECT-03** The reflection form shall ask the student to explain their submitted code, their AI usage, and whether they trusted, rejected, or modified AI-generated suggestions.
- **REQ-REFLECT-04** The reflection form shall have a time limit of 5 minutes.
- **REQ-REFLECT-05** If the student submits the reflection before the 5-minute limit expires, then the system shall store the reflection and mark the assessment attempt fully complete.
- **REQ-REFLECT-06** If the 5-minute reflection time limit expires, then the system shall automatically submit the assessment attempt and store any reflection text already entered.
- **REQ-REFLECT-07** If the reflection timer expires and the student entered no reflection text, then the reflection component score shall be 0.
- **REQ-REFLECT-08** If the reflection timer expires and the student entered partial reflection text, then the LLM shall evaluate the available text using the reflection rubric.
- **REQ-REFLECT-09** The system shall store the student reflection together with the submission or assessment attempt.
- **REQ-REFLECT-10** The system shall use an LLM to evaluate the student's reflection quality, solution understanding, AI usage quality, and critical handling of AI-generated output.
- **REQ-REFLECT-11** The LLM evaluation shall be shown in the administrator report.
- **REQ-REFLECT-12** The LLM evaluation shall be treated as decision support and shall not be the only basis for a final grading or hiring decision.

### 7.14 Process-Aware Scoring Rubric

- **REQ-SCORE-01** The system shall calculate a final process-aware score for administrator decision support.
- **REQ-SCORE-02** Each process-aware score component shall be scored on a 0-100 scale before weighting.
- **REQ-SCORE-03** The code correctness component shall be calculated from automated test results and stored submission outcomes.
- **REQ-SCORE-04** The AI usage quality component shall evaluate whether the student used hints strategically, selected appropriate hint levels, avoided excessive high-cost hints when lower-level hints were sufficient, and incorporated AI help in a way consistent with the final solution.
- **REQ-SCORE-05** The reflection understanding component shall evaluate whether the student accurately explains the submitted code, key algorithmic choices, complexity or tradeoffs where relevant, and the role of AI assistance.
- **REQ-SCORE-06** The critical AI judgment component shall evaluate how well the student handled AI Rescue suggestions, including whether the student accepted correct suggestions, rejected misleading suggestions, modified imperfect suggestions, and spent reasonable time reviewing the suggestion before deciding.
- **REQ-SCORE-07** If a student used AI Rescue during an assessment, the final process-aware score shall use these weights: code correctness 60%, AI usage quality 15%, reflection understanding 15%, and critical AI judgment 10%.
- **REQ-SCORE-08** If a student did not use AI Rescue during an assessment, the critical AI judgment component shall not be applied, and its 10% weight shall be reassigned to code correctness.
- **REQ-SCORE-09** If a student did not use AI Rescue during an assessment, the final process-aware score shall use these weights: code correctness 70%, AI usage quality 15%, and reflection understanding 15%.
- **REQ-SCORE-10** The LLM may support the evaluation of AI usage quality, reflection understanding, and critical AI judgment.
- **REQ-SCORE-11** The final process-aware score shall be shown as decision support for administrators and shall not represent a fully autonomous grading or hiring decision.
- **REQ-SCORE-12** The administrator report shall show the final process-aware score together with its component breakdown and short explanations for each component.

#### 7.14.1 Suggested 0-100 Rubric

The following rubric shall be used unless administrators configure a different rubric:

- **90-100 Excellent**: Strong evidence of independent understanding, appropriate AI use, clear reflection, and careful review of AI-generated suggestions.
- **70-89 Good**: Mostly correct understanding and reasonable AI use, with minor gaps or limited explanation.
- **50-69 Adequate**: Partial understanding or inconsistent AI use, but enough evidence to support the submitted solution.
- **25-49 Weak**: Limited understanding, vague reflection, overreliance on AI, or weak evidence of critical review.
- **0-24 Insufficient**: Empty, irrelevant, or clearly incorrect reflection; blind acceptance of misleading rescue output; or no meaningful evidence of understanding.

### 7.15 Reporting and Analytics

- **REQ-49** The system shall provide an administrator-facing report view.
- **REQ-50** When an administrator selects an assessment, the system shall show submission outcomes for participating students.
- **REQ-51** The report shall include at least student identifier, assessment identifier, submission status, code correctness score, test results, AI hint usage, AI credit usage, AI Rescue behavior, AI interaction summary, student reflection, LLM-based process evaluation, and process-aware score.
- **REQ-52** The system should provide simple aggregate statistics for each assessment, such as average score or completion counts.
- **REQ-53** The report should include a process-aware evaluation summary that distinguishes code correctness from AI-assisted process indicators.
- **REQ-54** The report shall not present an LLM-generated process evaluation as a fully autonomous final hiring decision.
- **REQ-55** The administrator report shall include full AI interaction details, including normal hint requests, AI responses, selected hint levels, credit costs, AI Rescue suggestions, rescue correctness labels, student rescue decisions, decision times, reflection text, LLM evaluation prompts or summarized evaluation context, LLM evaluation output, and process-aware score explanations.
- **REQ-56** Students shall only see their own released reports after an administrator releases final results for the assessment.
- **REQ-57** Released student reports shall include final scores, LLM evaluations, and short explanations, but shall not expose hidden test details or rescue correctness labels for other students.

### 7.16 AI Usage Tracking

- **REQ-58** The system shall record AI usage events during an assessment.
- **REQ-59** The system shall store at least the timestamp, user, assessment context, question context, and type of each AI usage event.
- **REQ-60** The system shall record the selected hint level, credit cost, rescue usage, rescue correctness state, rescue randomness outcome, student decision on rescue output, and rescue decision time where applicable.
- **REQ-61** The system shall make AI usage summaries visible to administrators.
- **REQ-62** The system shall store LLM-generated evaluations with enough context to allow administrator review.

### 7.17 Assessment Attempt and Identity Scope

- **REQ-63** The backend shall identify the current user from the authentication context, such as JWT or another secure token, rather than relying on a frontend-supplied user identifier.
- **REQ-64** When a student starts or resumes an assessment, the backend shall resolve the active assessment attempt from the authenticated user and assessment identifier.
- **REQ-65** The frontend shall not be responsible for creating, storing, trusting, or sending a real `session_id`; backend-connected assessment flows shall use assessment-scoped APIs where the backend resolves the active attempt from authentication context.
- **REQ-66** Workspace code shall be scoped to the authenticated user, assessment, and question so that each student sees only their own in-progress code.

Terminology clarification: older architecture material may use `session_id` for the assessment attempt identifier. In the current requirements, that concept is backend-owned and should be treated as an assessment attempt resolved from authenticated user and assessment ID. The frontend must not create, persist, send, or trust this identifier as authoritative state.

## 8. Non-Functional Requirements

### 8.1 Security

- **NFR-01** The platform shall enforce authentication and role-based authorization.
- **NFR-02** The platform shall execute student code only in sandboxed environments.
- **NFR-03** The platform shall protect stored user, submission, and AI interaction data against unauthorized access.
- **NFR-04** The platform shall apply secure handling for AI-related data and secrets used for external AI APIs.
- **NFR-05** The platform shall prevent students from viewing hidden correctness labels for AI Rescue suggestions during the assessment.
- **NFR-06** The platform shall prevent student-facing views from exposing hidden test inputs, hidden expected outputs, grading implementation, or other students' reports.

### 8.2 Usability

- **NFR-07** The student workflow shall be simple enough to complete an assessment without local environment setup.
- **NFR-08** The administrator workflow shall clearly separate authoring, monitoring, report review, and report release activities.
- **NFR-09** The interface should remain understandable for first-time users.
- **NFR-10** AI hint levels, remaining AI credits, and remaining AI Rescue chances should be visible and understandable during the assessment.

### 8.3 Reliability

- **NFR-11** The system should save submissions and evaluation results reliably.
- **NFR-12** If execution fails unexpectedly, then the platform should fail gracefully and preserve already saved assessment data.
- **NFR-13** If an LLM service fails, then the platform should preserve the assessment attempt and show an actionable error message instead of losing student work.
- **NFR-14** If report release fails, then unreleased student reports should remain hidden until an administrator successfully releases them.

### 8.4 Performance

- **NFR-15** The coding workspace should remain responsive during normal assessment usage.
- **NFR-16** Execution feedback should be returned within a reasonable time for normal-sized assessment tasks.
- **NFR-17** Report pages should load within a reasonable time for normal course or interview cohorts.
- **NFR-18** LLM-based task generation, AI assistance, rescue generation, and reflection evaluation should return within a reasonable time or fail gracefully.

### 8.5 Maintainability and Extensibility

- **NFR-19** The system should be modular enough to separate authentication, assessment management, execution, AI integration, rescue generation, reflection evaluation, scoring, and reporting concerns.
- **NFR-20** The project should be structured so that new languages or AI features can be added later with limited impact on existing modules.
- **NFR-21** AI scoring rules, hint costs, rescue count, and rescue correctness probability should be configurable rather than hard-coded where feasible.

### 8.6 Auditability and Traceability

- **NFR-22** Requirements, implementation prompts, code changes, tests, and review notes should be traceable through requirement IDs where practical.
- **NFR-23** Changes to requirements should be versioned alongside implementation artifacts.
- **NFR-24** AI-generated evaluations should be stored with enough context to allow administrator review.
- **NFR-25** AI Rescue randomness outcomes should be logged for auditability.

## 9. Constraints

- The system must support at least student and administrator roles.
- The system must support Python and JavaScript as student coding languages in the first student-facing implementation.
- The full system must provide sandboxed code execution.
- The system must provide performance reporting for administrators.
- AI task generation, normal AI hints, AI Rescue, and reflection evaluation require access to an external LLM API or integrated model service.
- AI Rescue correctness labels must not be visible to students during the assessment.
- Unrestricted free-form AI chat must not be available during assessments.
- Administrators control when final student reports are released.
- The project should remain implementable within a university group-project scope.

---

## 10. Assumptions

- Users have access to a modern web browser and stable internet connection.
- The platform can access a secure runtime or container-based execution environment.
- AI assistance is available through an external API or integrated model service.
- The first version is an MVP rather than a production-scale enterprise platform.
- Administrators review LLM-generated tasks before publication.
- LLM-based process evaluation is used as decision support, not as a fully autonomous final decision.
- AI credits are scoped per question.
- AI Rescue chances are scoped per assessment.

## 11. Acceptance Summary by Capability

### A. Assessment Authoring

Accepted when an administrator can create an assessment, add questions, configure test cases, configure AI feature settings, configure assessment-level AI Rescue probability, and activate the assessment.

### B. LLM Task Generation

Accepted when an administrator can generate a draft coding task with an LLM, review and edit the generated content and test cases, and accept, reject, regenerate, or add the task to an assessment.

### C. Student Coding Workflow

Accepted when a student can open an assessment, read a question, write code, run code, use enabled structured AI features, and submit a solution.

### D. Safe Execution

Accepted when code runs in an isolated environment and failing executions are reported without compromising the platform.

### E. Structured AI Support and Logging

Accepted when AI hint levels, AI credit usage, and AI Rescue behavior are recorded and reviewable for enabled assessments.

### F. Reflection and LLM Process Evaluation

Accepted when students can submit reflections after their solutions and administrators can view LLM-based evaluation of understanding, AI usage quality, and critical handling of AI output.

### G. Process-Aware Scoring

Accepted when administrators can view code correctness, AI usage quality, reflection understanding, critical AI judgment where applicable, a final process-aware score, and short explanations for each component.

### H. Controlled Student Report Release

Accepted when administrators can release final reports and students can view only their own released final scores, LLM evaluations, and short explanations.

### I. Reporting

Accepted when administrators can review per-student results, code correctness, test results, AI credit usage, rescue behavior, full AI interaction logs, student reflections, LLM-based process evaluations, and process-aware scores for a selected assessment.

- Implementation prompts, code changes, tests, and review notes should reference one or more requirement IDs where practical.
- Verification artifacts or test cases should reference the requirement or requirements they validate.
- UI implementation should use `ui-style-reference.md` where relevant, but that file is a visual style reference only and must not override this specification.

## 13. Open Questions and Resolved Decisions

- Resolved: students see final scores, LLM evaluations, and short explanations only after an administrator releases results.
- Resolved: process score components use a 0-100 scale.
- Resolved: AI credits are per question.
- Resolved: AI Rescue provides 4 chances per assessment.
- Resolved: AI Rescue correctness probability is configured at assessment level.
- Resolved: the default AI Rescue correctness probability is 50% correct and 50% misleading.
- Resolved: misleading AI Rescue suggestions are normal logically wrong solutions and must not be unsafe or platform-breaking.
- Resolved: if the 5-minute reflection timer expires and the student wrote nothing, the reflection component score is 0.
- Resolved: if the 5-minute reflection timer expires and the student wrote partial text, the LLM evaluates the available text using the reflection rubric.
- Resolved: unrestricted free-form AI chat is not available during assessments.
- Resolved: administrators can view full AI interaction details.
- Resolved: LLM-generated tasks include draft test cases, and administrators review them before publication.
- Resolved: authentication identifies the user, and the backend resolves the active assessment attempt from authenticated user and assessment ID.
- Resolved: the frontend does not manage a real `session_id`.

## 15. Summary

This requirements document defines a structured, testable basis for building an AI-assisted coding assessment platform. It emphasizes stakeholder intent, explicit scope boundaries, EARS-style requirement statements, traceability, and implementation prompts that can reference requirement IDs directly.

The extended feature set makes the project more distinct from a standard online judge by evaluating not only whether submitted code works, but also how students use AI, how strategically they spend AI credits, whether they critically evaluate AI Rescue suggestions, and whether they understand their final solution.
